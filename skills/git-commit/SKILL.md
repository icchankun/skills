# /git-commit スキル

gitの差分や会話の文脈をもとに、[Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) に従ったコミットメッセージを生成し、コミットを作成する

## 手順

### ステップ1: 現状把握

以下のコマンドを**並列**で実行:

- `git status`（`-uall` は使わない）
- `git diff --staged`
- `git diff`
- `git log --oneline -5`

直近のコミット履歴からメッセージのスタイルや粒度を把握する

### ステップ2: ステージング判定

ステージ済みファイルがない場合:
- 会話の文脈・実装状況からステージすべきファイルを判断する
- `.env`・認証情報等が含まれる場合は警告する

ステージ済みファイルがある場合はそのまま続行
複数の論理的変更が混在する場合は分割を提案する

### ステップ3: コミットメッセージ生成

変更理由が不明な場合はユーザーに質問する
後述の「コミットメッセージ規則」に従って生成する

### ステップ4: ユーザー確認（1回のみ）

以下の内容をまとめて1回だけ確認する:

- **ステージ対象**: ステージするファイル一覧（既にステージ済みの場合はその旨）
- **コミットメッセージ**: 生成したメッセージ全文

承認後、以下を順次実行:
1. ファイルのステージ（必要な場合）: `git add` で個別にステージ
2. コミット実行（HEREDOC形式）:

```bash
git commit -m "$(cat <<'EOF'
コミットメッセージ1行目

本文（必要な場合）
EOF
)"
```

### ステップ5: hook失敗時の対応

pre-commit hookが失敗した場合:

1. エラー内容を分析し、問題を修正する
2. 修正ファイルを再ステージ
3. **新規コミット**を作成する（hookの失敗時はコミットが未作成のため、`--amend` は前のコミットを書き換えてしまう）

## コミットメッセージ規則

### 記述ルール

- 句点（。）は使わず、文の区切りでは改行する
- 本文は敬体で記述する
- **1行目**・**「なぜ必要なのか？」**・**「変更内容」**にはコード要素（クラス名・メソッド名・変数名）を含めず、コードを見たことがない人でも分かる文章で説明する
- **1行目**: 体言止め、50文字以内（最大72文字）　末尾に句読点・絵文字は付けない
- 技術用語は公式表記を使う（GitHub, RuboCop, TypeScript, Next.js, PostgreSQL, nginx 等）

### 変更理由（why）の記載

変更内容（what）は `git diff` で確認できるため、コミットメッセージには変更理由（why）を記載する
diffの情報より**会話の文脈**を優先し、以下を読み取る:

- 何が問題だったのか（背景・課題）
- このコミットでどう改善されるのか（効果・目的）

typo修正など自明な変更を除き、変更が必要だった理由を記載する

「不要」「不適切」「古い」などの主観的な判断を含む場合は、客観的な根拠（検索結果、意思決定の経緯、技術的事実、エラーの証拠、数値データ等）を添える

### テンプレート

本文が必要な場合:

```
型: 変更内容の要約（体言止め）

## 🎯 なぜ必要なのか？

~のためです（背景・課題の説明）。~のため、~します（課題と、それに対する対応方針の説明）。
箇条書きにせず散文で記述する。

## 🔧 変更内容

### [絵文字] [見出し]

[詳細 - 敬体　コードを見たことがない人でも分かるように説明する]

## 📎 参考

- [タイトル](URL)
```

- 参考セクションは根拠リンクがある場合のみ追加する
- 見出しの後には必ず空行を入れる

本文を省略できるのは「なぜ」が自明な場合のみ（typo修正、lint修正等）
「不要コード削除」は省略不可

### Conventional Commits prefix

参考: [Conventional Commits v1.0.0](https://www.conventionalcommits.org/en/v1.0.0/) / [Angular Commit Message Guidelines](https://github.com/angular/angular/blob/22b96b9/CONTRIBUTING.md#-commit-message-guidelines)

prefixは原則必須:

| prefix | 用途 |
|--------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `chore` | リリースノートに書く必要のない作業 |
| `refactor` | バグ修正でも新機能追加でもないコード変更 |
| `perf` | パフォーマンス改善 |
| `style` | コードの意味に影響しない変更 |
| `test` | テストの追加・修正 |
| `docs` | ドキュメントのみの変更 |
| `ci` | CI設定の変更 |
| `build` | ビルドシステムや外部依存関係の変更 |
| `remove` | コードやファイルの削除 |

破壊的変更がある場合は型の後に `!` を付ける（例: `feat!: APIレスポンスの形式を変更`）

### 絵文字

詳細は [gitmoji.dev](https://gitmoji.dev) 参照

見出し系: 🎯 目的・背景 / 🤔 課題・問題 / 🔧 変更内容 / 📎 参考

変更種別系: ✨ 新機能 / 🐛 バグ修正 / 🚑️ 緊急修正 / ♻️ リファクタ / 🔥 削除 / ⚡️ パフォーマンス / 🔒️ セキュリティ / 💄 UI/UX / ✅ テスト / 📝 ドキュメント / 🌐 国際化 / ⬆️ 依存関係更新 / 👷 CI/CD / 💥 破壊的変更

## 禁止事項

- `git push` はユーザーが明示的に依頼した場合のみ実行する
- `--no-verify` は使わない
- `.env`・認証情報・シークレットファイルのコミットは警告する
