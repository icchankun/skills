# /github-pr スキル

gitのコミット情報の解析からdraft PR作成まで一気通貫で行う。
PRタイトルとdescriptionをコミットメッセージから自動生成することで、手作業でのコピペや整形を不要にする。

## 手順

### ステップ1: 現状を把握する

以下を**並列**で実行し、PRに含まれるコミットと変更内容を把握する:

```bash
git branch --show-current
git log --oneline develop..HEAD
git log develop..HEAD --reverse --format='%B---COMMIT_SEPARATOR---'
git diff develop..HEAD --name-only
```

**中断条件** — 以下のいずれかに該当する場合は処理を止め、理由を伝える:

- 現ブランチが `develop` または `main` である — 保護ブランチへの直接pushが発生するため
- develop上にコミットがない — PRに含める変更がないため

### ステップ2: PRタイトルを組み立てる

PRタイトルは `[チケットID][サービス名] サマリー` の形式にする。

#### チケットIDの抽出

ブランチ名からチケットIDパターンを探す。代表的なパターン:

| パターン | 例 | 抽出結果 |
|----------|------|----------|
| `[A-Z]+-\d+`（JIRA / Notion 等） | `feat/PROJ-123-add-auth` | `[PROJ-123]` |
| `#\d+`（GitHub Issues） | `fix/#42-login-error` | `[#42]` |
| 該当なし | `fix/login-error` | 省略 |

#### サービス名の推定

変更ファイルのパスやクラスの名前空間から、この変更がどの機能領域に属するかを推定する:

1. `git diff develop..HEAD --name-only` の `app/` 配下のパスを見る
2. モデル・コントローラ等のディレクトリ名（名前空間）の出現頻度を集計する
3. 最も多い名前空間からチームが普段PRタイトルで使うサービス名を判断する

複数領域にまたがる場合や汎用的な名前空間しかない場合は省略する。

#### サマリーの抽出

最初のコミット（`--reverse` で最も古いもの）の1行目から、Conventional Commitsのprefix（`feat:`, `fix:`, `refactor:` 等）を除いた部分をサマリーとする。

#### 組み立てパターン

要素の有無に応じて自然に組み立てる:

```
[PROJ-123][認証] ログイン時のトークン検証を追加
[PROJ-123] 依存パッケージのアップデート
[認証] ログイン時のエラーハンドリングを修正
ログイン時のエラーハンドリングを修正
```

### ステップ3: PR descriptionを作成する

プロジェクトに `.github/PULL_REQUEST_TEMPLATE.md` がある場合はそれを読み込み、ベースにする。

コミットメッセージの本文（3行目以降）を、テンプレートの冒頭コメント・ヘッダ行の直後・最初の `##` 見出しの手前に挿入する。テンプレートの他のセクション（`## 動作確認`、`## チェックリスト` 等）はそのまま残す。

**挿入イメージ（テンプレートがある場合）:**

```markdown
※リリースフローについては [deploy.md](...) を確認してください。

[コミットメッセージの3行目以降をここに挿入]

## 動作確認

...
```

テンプレートがない場合は、コミットメッセージの本文（3行目以降）をそのまま description とする。

**複数コミットがある場合:**

- 「なぜ必要なのか？」に相当する背景・課題の記述は、各コミットの内容が自然につながるよう文章としてまとめ直す
- 「変更内容」に相当する箇条書きも、コミット単位ではなく変更の意味単位でまとめる

### ステップ4: ユーザーに確認する

以下を提示して承認を得る:

- **PRタイトル**
- **PR description**（全文）
- **ベースブランチ**: develop

ユーザーがタイトルやdescriptionの修正を求めた場合は反映してから進む。

### ステップ5: rebase & push & PR作成

承認後、以下を順に実行する:

1. developの最新をrebase:
   ```bash
   git pull --rebase origin develop
   ```
   コンフリクトが発生した場合はユーザーに伝えて処理を中断する。自動でコンフリクト解消は行わない。

2. リモートにpush（未pushの場合）:
   ```bash
   git push -u origin HEAD
   ```

3. draft PRを作成:
   ```bash
   gh pr create --draft --base develop --title "タイトル" --body "$(cat <<'EOF'
   description全文
   EOF
   )"
   ```

4. 作成されたPRのURLをユーザーに表示する。

## 禁止事項

- ユーザーの承認なしに `git push` や `gh pr create` を実行しない — 共有リポジトリへの操作は確認が必須
- `develop` や `main` ブランチ上では絶対にpushしない — 保護ブランチへの直接pushは事故のもと
- コミットの作成・変更はしない — それは `/git-commit` の役割
